## Мета заняття
Навчитися працювати з git, створювати гілки, робити merge та вирішувати конфлікти.

1. Основи Git
Яка різниця між Git та GitHub? Git — це інструмент (програма) для контролю версій, встановлений на вашому комп'ютері. GitHub — це веб-сервіс (хостинг), де зберігаються Git-репозиторії в інтернеті.

Що означає "розподілена" система контролю версій? Це означає, що кожен розробник має на своєму комп'ютері повну копію історії проєкту, а не лише файли поточної версії.

Яка різниця між git init та git clone? git init створює новий, порожній репозиторій у поточній папці. git clone копіює існуючий репозиторій з віддаленого сервера.

2. Staging Area та Commits
Що таке Staging Area і навіщо вона потрібна? Це проміжна зона (буфер) між робочою директорією та репозиторієм. Вона дозволяє вибрати конкретні зміни, які потраплять у наступний коміт, не додаючи туди все підряд.

Яка різниця між git add . та git add -A? В сучасних версіях Git різниця мінімальна, але технічно: git add . додає зміни в поточній директорії та підпапках, а git add -A додає зміни у всьому дереві проєкту (навіть якщо ви знаходитесь у підпапці).

Як написати гарне commit message? Використовуйте імператив ("Add feature", а не "Added feature"). Пишіть короткий заголовок (до 50 символів) і, за потреби, детальний опис через порожній рядок.

Як виправити останній коміт? Використовуйте команду git commit --amend. Це дозволяє змінити повідомлення або додати забуті файли до останнього коміту.

3. Гілки та злиття
Що таке гілка в Git і чому вони "легковажні"? Гілка — це просто рухомий вказівник на певний коміт. "Легковажна", бо створення гілки займає мить і майже не займає місця (це файл розміром 41 байт), на відміну від копіювання файлів.

Яка різниця між git merge та git rebase? merge зберігає історію, створюючи спеціальний "merge commit" (з'єднує гілки). rebase переписує історію, накладаючи ваші зміни поверх цільової гілки (робить історію лінійною).

Що таке fast-forward merge? Це злиття, коли немає розгалуження історії. Git просто пересуває вказівник головної гілки вперед на місце вашої нової гілки.

Коли виникають merge конфлікти і як їх уникнути? Виникають, коли в двох гілках змінено один і той самий рядок коду. Уникнути: робити маленькі коміти, часто оновлювати свою гілку (git pull) та спілкуватися з командою.

Чому не можна робити rebase публічних комітів? Бо rebase змінює хеші комітів (переписує історію). Якщо хтось інший вже завантажив ці коміти, у них виникнуть проблеми з синхронізацією.

4. Віддалені репозиторії
Що таке origin? Це стандартний псевдонім (ім'я) для посилання на віддалений репозиторій, з якого ви клонували проєкт.

Яка різниця між git fetch та git pull? git fetch завантажує зміни з сервера, але не змінює ваш код. git pull робить fetch, а потім автоматично робить merge у вашу поточну гілку.

Що робить git push --force і чому це небезпечно? Він примусово перезаписує історію на сервері вашою локальною версією. Небезпечно, бо це може видалити роботу інших розробників, яку вони встигли запушити.

Як видалити віддалену гілку? git push origin --delete <назва_гілки>.

5. Скасування змін
Яка різниця між git reset --soft, --mixed та --hard?

--soft: Відкочує коміт, але залишає зміни в Staging area (готові до коміту).

--mixed (за замовчуванням): Відкочує коміт, зміни залишаються в робочій папці (не застейджені).

--hard: Повністю знищує зміни і повертає стан до вибраного коміту.

Яка різниця між git reset та git revert? reset рухає час назад (видаляє історію). revert створює новий коміт, який дзеркально скасовує зміни попереднього (безпечно для історії).

Як скасувати git add (видалити файл зі staging)? git restore --staged <файл> (або git reset HEAD <файл>).

Як повернути видалений файл? git restore <файл> (якщо не було коміту) або git checkout <коміт> -- <файл> (щоб витягнути з історії).

6. Безпека
Що не повинно потрапляти в Git репозиторій? Паролі, API ключі, папка node_modules (та інші залежності), бінарні файли збірки (.exe, /bin), системні файли (.DS_Store).

Що робити якщо випадково закомітили пароль?

Негайно змінити пароль у сервісі.

Видалити пароль з історії Git (використовуючи git filter-branch або інструмент BFG Repo-Cleaner), а не просто новим комітом.

HTTPS vs SSH для GitHub — що краще? SSH краще, бо не треба постійно вводити пароль/токен, і це безпечніше.

Що таке Personal Access Token і навіщо він потрібен? Це довгий код, який використовується замість пароля при доступі через HTTPS, оскільки GitHub заборонив аутентифікацію звичайним паролем акаунту для операцій у терміналі.

7. GitHub
Що таке Pull Request (PR)? Це запит на злиття змін з вашої гілки в основну гілку. Це місце для перевірки коду (Code Review) та обговорення перед влиттям.

Яка різниця між Issues та Pull Requests? Issues — це задачі, баги або обговорення (текст). Pull Requests — це пропозиція конкретних змін у коді.

Що таке Fork і навіщо він потрібен? Це копія чужого репозиторію на вашому акаунті. Потрібен, щоб вносити зміни в чужий проєкт без прямих прав доступу до нього (Open Source).

Що таке GitHub Actions? Це інструмент для автоматизації (CI/CD). Наприклад, автоматичний запуск тестів або деплой сайту при кожному пуші.

8. Best Practices
Як часто треба робити коміти? Часто. Кожна логічно завершена мікро-задача має бути окремим комітом.

Скільки часу має жити feature branch? Якнайменше. Від кількох годин до пари днів. Довгоживучі гілки призводять до складних конфліктів ("merge hell").

Як організувати git workflow в команді? Домовитись про єдиний стиль (наприклад, Git Flow або GitHub Flow). Завжди робити pull перед початком роботи. Не пушити в main напряму.

Чи треба комітити node_modules або інші залежності? Ніколи. Для цього є файл .gitignore. Залежності встановлюються командою npm install на основі package.json.
